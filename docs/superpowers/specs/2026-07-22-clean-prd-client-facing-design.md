# Clean, client-facing prd.md

## Problem

The PM flagged that the generated `prd.md` reads as an internal pipeline artifact rather
than a document handed to a client. Specifically:

- A `> Generated by sandwich/order · mode: X · date` banner
- A "Project State" table exposing internal fields (`phase`, `hasExistingCodebase`,
  `orderSource`)
- Every actor/feature/integration/constraint bullet tagged with
  `` `[stated]`/`[discussed]`/`[inferred]`/`[assumed]` ``
- Module status badges (`planned`/`exists`/`partial`/`broken`)
- A `## Changes since last run` changelog block on every regeneration
- A closing note addressed to the task-breakdown pipeline, not the reader

`prd.md` is handed to clients directly, so none of this pipeline plumbing can remain
visible in it.

## Constraint: confidence data is load-bearing downstream

The confidence tags aren't decorative. `prep/agents/01-extract-features.md` reads
`prd.md`'s raw text and derives each extracted feature's `confidence` by pattern-matching
the inline `[stated]/[discussed]/[inferred]/[assumed]` tags (see its "Confidence markers"
section). Stripping the tags from `prd.md` without another path for that signal would
silently degrade `/prep`'s scoring.

The fix: `order/lib/order-lib.ts::writeOrderArtifact` already writes a parallel structured
`prd.json` next to `prd.md` on every run (same data, before rendering). `wireframe.workflow.ts`
already establishes the precedent of a downstream package importing
`readOrderDocs` from `order/lib/order-lib.ts` directly instead of re-parsing markdown.
`/prep` will follow the same pattern for confidence data.

## Design

### 1. `order/lib/order-render.ts` — `renderPrd`

Rewrite to produce a clean, prose-first document:

```
# {ProjectName} — Product Requirements Document

## Overview
{2-3 sentence prose; folds in existing-codebase / greenfield context naturally
 instead of a Phase/hasExistingCodebase/orderSource table}

## Actors
- {name} — {role}

## Modules & Features
### {module name}
{description}
- {feature text}

## Integrations
- {text}

## Constraints
- {text}

## Stakeholders
- {name} — {role}

## Timeline
{text or "Not specified"}

## Open Questions
{N} items need your input — see client-questions.md
```

Removed entirely from the rendered output:
- The generator banner line (mode/date)
- The Project State table
- All `` `[confidence]` `` tags on actors/features/integrations/constraints
- Module `status` badges
- The `withChangelog` call for PRD rendering (no changelog section in `prd.md`)
- The closing "For task breakdown pipeline" note

No schema change: `PrdDocSchema` (`order/lib/order-schemas.ts`) is untouched. All of this
data (`mode`, `projectState`, `confidence`, module `status`) still exists in full in
`prd.json` — only the markdown projection changes. `02-write-prd.md` (the generator agent)
is untouched; it keeps producing the same structured JSON.

Scope: only `renderPrd` changes. `renderUserFlows`, `renderTechNotes`, and
`renderClientQuestions` (and their changelog behavior) are unaffected — the PM's complaint
was specific to the PRD.

### 2. `/prep` reads `prd.json` for confidence instead of scraping `prd.md`

Changes confined to the `prep` package:

- `prep/workflow/prep.workflow.ts`: alongside the existing `readBriefArtifacts` text read,
  load the structured doc with `readOrderDocs(projectRoot)` (imported from
  `../../order/lib/order-lib.js`, mirroring `wireframe.workflow.ts:18`). Pass the resulting
  `prdDoc` into the Phase 3 extraction context as a new `prdDoc` field, alongside the
  existing `briefArtifacts.prd` text.
- `prep/agents/01-extract-features.md`: update the "Input" section to document `prdDoc`
  (structured: `modules[].features[].confidence`, `actors[].confidence`, etc.). Replace the
  "Confidence markers" section (which taught the agent to scan for bracket tags in prose)
  with an instruction to read `confidence` directly off the matching item in `prdDoc`.
  `source.line` citation instructions are unchanged — the agent still locates the feature's
  text inside the plain-prose `prd.md` string to cite a line number, it just no longer needs
  that string to carry the confidence signal too.
- Staleness hashing (`hashFile(briefArtifacts.prd)` in `prep.workflow.ts`) is unchanged —
  still hashes the rendered `prd.md` text. (Out of scope: switching this to hash `prd.json`
  is a separate concern from the PM's PRD-readability complaint and isn't needed to fix it.)

Untouched: `user-flows.md`, `technical-notes.md`, `client-questions.md` rendering and
consumption — this fix is scoped to the PRD only.

## Testing

- `order/lib/order-render.ts`: existing/new unit tests assert `renderPrd`'s output contains
  none of the removed elements (no `[stated]` etc., no "Project State", no "Generated by",
  no "Changes since last run") and does include the plain prose sections.
- `prep`: extend the extraction self-check (`prep/lib/validation.selfcheck.ts` /
  `prep/lib/spec.selfcheck.ts` as appropriate) to cover a `prdDoc`-driven confidence case —
  a feature whose confidence comes from `prdDoc` even though the matching `prd.md` prose
  string has no tag in it.
- Manual: run `/order` end-to-end on a sample input, confirm `prd.md` reads cleanly, then
  run `/prep` against it and confirm extracted features still carry sensible `confidence`
  values and `feature-queue.md` still shows `Source: prd.md:<line>` citations.

## Out of scope

- Changes to `user-flows.md`, `technical-notes.md`, `client-questions.md` rendering.
- Switching `/prep`'s staleness hash from `prd.md` text to `prd.json`.
- Any change to `02-write-prd.md` (the PRD-generating agent) — it already produces the full
  structured document; only the markdown projection of it changes.
