# Per-Feature Spec Files — Design

**Date:** 2026-07-02
**Status:** Approved (design review with Ria)

## Problem

`/prep` produces a useful prioritized queue (`feature-queue.md`) but no
per-feature artifact a developer can pick up and start from. The old `/recipe`
step that generated specs was removed (commit `a9d6a7f`) because it was a weak
spec step with no human gate — its replacement is Superpowers brainstorming,
but the hand-off to it is implicit: nothing in the output says "open this file,
start here."

Two secondary reliability problems surfaced during investigation:

1. **Silent incompleteness.** In real runs (aresphi), the LLM skipped writing
   `decisions.json` even though the journal recorded decisions — and nothing
   noticed. There is no end-of-run completeness check.
2. **Runtime reality.** Both Pi and Claude Code execute `/order` and `/prep` as
   *skills* (LLM follows SKILL.md prose). The `*.workflow.ts` files are dead
   code in both runtimes: `workflows` is not a recognized `plugin.json` field
   in Claude Code (verified against official docs), and Pi's extensions only
   register skill paths + the registry write gate. The only deterministic
   pieces that actually run are the `scripts/render.ts` files, invoked by the
   LLM via SKILL.md instructions. **This design deliberately extends that
   proven pattern instead of introducing runtime-specific execution hooks.**

## Goals

- After `/prep`, every active feature has its own spec file a dev can open and
  hand to `superpowers:brainstorming`.
- Spec format is deterministic: LLM supplies content as validated JSON, code
  renders the markdown. Same format every run, every runtime.
- Each spec carries an acceptance-criteria checklist that gets checked off as
  implementation proves each criterion.
- `/prep` fails loudly when any expected artifact is missing (closes the
  aresphi-class bug).
- `/status` audits completeness, not just lifecycle counts.

## Non-Goals (explicit)

- **No task breakdown, estimates, or file lists in specs.** That was the
  weakness that killed `/recipe`. Implementation planning belongs to
  Superpowers brainstorming/writing-plans, which have codebase access.
- **No runtime-specific execution** (Claude Code `UserPromptSubmit` hooks, Pi
  `registerCommand`). Investigated and rejected: high effort, per-runtime
  divergence, and it doesn't address the actual failure mode (unverified
  completeness), since spec *content* is LLM judgment either way.
- **No checklist→lifecycle sync.** Checking all ACs does not auto-update
  `features.json` lifecycle. The format enables this later; out of scope now.
- **No gate-approval enforcement.** `queueApproved` staying false is a team
  process issue, not a code issue.

## Design

### 1. Spec artifact pair: `docs/sandwich/specs/F-XXX.json` + `.md`

JSON is the source of truth (LLM-written, zod-validated). MD is a
deterministic projection, overwritten on every render — same philosophy as
`feature-queue.md` ("never read back as state"). Checking off an AC means
setting `done: true` in the JSON and re-rendering.

Schema (new file `prep/lib/spec-schema.ts`):

```ts
const AcceptanceCriterionSchema = z.object({
  id: z.string().regex(/^AC\d+$/),
  text: z.string().min(1),
  done: z.boolean().default(false),
});

export const FeatureSpecSchema = z.object({
  featureId: z.string().regex(/^F-\d{3}$/),
  title: z.string().min(1),
  module: z.string().min(1),
  description: z.string().min(1),
  scope: z.object({
    inScope: z.array(z.string()).min(1),
    outOfScope: z.array(z.string()), // may be empty
  }),
  acceptanceCriteria: z.array(AcceptanceCriterionSchema).min(1),
  dependsOn: z.array(z.string()),
  source: z.object({ file: z.string(), lines: z.string().optional() }),
});
```

Rendered MD layout (fixed by code):

```
# F-001: OTP Verification Flow

> **Module:** auth · **Priority:** 27/100 · **Depends on:** — · **Source:** prd.md:31-33

<description>

## Scope
**In:** … / **Out:** …

## Acceptance Criteria
- [ ] **AC1** — …

---
## Mulai kerja
1. Jalankan superpowers:brainstorming dengan file ini sebagai starting point …
2. Setelah implementasi, centang tiap AC yang terbukti jalan
   (update `done` di F-001.json, lalu re-run render).

_Generated by /prep — edit F-001.json, bukan file ini (akan di-overwrite)._
```

Priority in the header is pulled from the registry (`effectivePriority`),
joined by `featureId` — the spec JSON never carries priority.

### 2. `prep/scripts/render-specs.ts` (new, deterministic)

Mirrors `prep/scripts/render.ts`:

- Args: `[project-root]` (defaults to cwd).
- Reads every `docs/sandwich/specs/*.json`, validates against
  `FeatureSpecSchema`, joins registry features for priority.
- Renders each to `specs/F-XXX.md`.
- Exit 1 listing *all* validation errors (file + field + message); exit 0 with
  one `✓` line per file written.

### 3. `prep/scripts/verify-complete.ts` (new, deterministic)

End-of-`/prep` completeness gate. Checks, in order:

1. Registry files exist and parse: `project.json`, `features.json`,
   `questions.json`.
2. Every `decision-recorded` target in `journal.jsonl` exists in
   `decisions.json` (this is the aresphi bug).
3. Every active feature (lifecycle not `done`/`rejected`) has
   `specs/F-XXX.json` that validates, and `specs/F-XXX.md`.
4. `docs/sandwich/feature-queue.md` exists.

Any failure → exit 1 with a precise, actionable list ("F-004 has no spec —
write docs/sandwich/specs/F-004.json and re-run render-specs"). SKILL.md
instructs the LLM to fix and re-run until exit 0.

### 4. `specRef` wiring (re-activates existing drift detection)

When `/prep` writes a spec, the feature's `specRef` in `features.json` is set
to `specs/F-XXX.json`. `registry-lib.ts` already uses `specRef` for stale-spec
detection (ripple report, `flags.stale`) — this field has been orphaned since
`/recipe` was removed; specs existing again makes it meaningful.

### 5. `feature-queue.md` slims down

- Queue table gains a `Spec` column linking to `specs/F-XXX.md`.
- The per-feature `### F-XXX` detail blocks are removed (content now lives in
  the spec files). `Changes Since Last Run`, `Recommendation`, and `History`
  sections stay.
- Change lands in `renderFeatureQueue()` (`registry/registry-io.ts`).

### 6. `/status` becomes an auditor

`renderStatus()` (`registry/registry-io.ts`) stays pure; it gains inputs for
audit findings and renders new lines under `Awaiting you`:

- "N active feature(s) missing specs: F-004, F-007"
- "journal records decision D3 but decisions.json doesn't have it"

Since `/status` is also a prose skill in both runtimes, a new thin script
`prep/scripts/status.ts` (pattern: `render.ts`) reads the registry + specs
dir, calls `renderStatus()`, prints. Status SKILL.md is updated to run it
instead of hand-assembling the dashboard.

### 7. SKILL.md updates (`prep/skills/prep/SKILL.md`)

- New step after registry writes: "write `docs/sandwich/specs/F-XXX.json` for
  every active feature" with the exact schema embedded (same convention as the
  embedded registry schemas).
- Renderer step now runs both `render.ts` and `render-specs.ts`.
- Final step: run `verify-complete.ts`; on exit 1, fix the listed items and
  re-run until clean.

### 8. Cleanup

- Delete dead `writeSpec()` + `Spec` interface from `prep/lib/prep-lib.ts`
  (zero callers since `/recipe` removal; replaced by this design).
- Selfcheck coverage: new `prep/lib/spec.selfcheck.ts` (or extension of
  `validation.selfcheck.ts`) covering schema validation, render output shape,
  and verify-complete pass/fail cases. Wire into `package.json` test script.
- README pipeline section updated to show the spec hand-off.

## Resulting layout (consumer project)

```
docs/sandwich/
├── prd.md / user-flows.md / technical-notes.md / client-questions.md  (+ .json twins)
├── feature-queue.md          ← summary: priorities + links to specs
└── specs/
    ├── F-001.json            ← source of truth (LLM-written, zod-validated)
    ├── F-001.md              ← deterministic projection, dev starting point
    └── …
.sandwich/registry/           ← unchanged (project/features/questions/decisions/journal)
```

## Error handling

- `render-specs.ts` / `verify-complete.ts` never partially succeed silently:
  all errors are collected and printed, exit code is the contract.
- The pi-gate is untouched; registry writes remain gated. Spec files live
  outside `.sandwich/registry/` so the gate does not apply to them —
  `render-specs.ts` validation is their gate.
- A spec JSON the LLM writes for a feature that doesn't exist in the registry
  is a verify-complete error (orphan spec), not a crash.

## Testing

- Selfcheck (deterministic, no LLM): schema accepts/rejects fixtures; renderer
  output contains expected sections and checklist syntax; verify-complete
  passes on a complete fixture tree and fails with the right messages on each
  class of missing artifact.
- Manual: run `/prep` on a real project (digitalhub-3) via Pi; confirm specs
  appear, queue links resolve, verify-complete exits 0, and `/status` reports
  clean.

## Key decisions log

| Decision | Choice | Why |
|---|---|---|
| Checklist source of truth | JSON (`done` field), MD is projection | Consistent with registry philosophy; render stays overwrite-safe |
| Task breakdown in specs | No | That weakness killed /recipe; brainstorming owns it |
| Determinism approach | Extend LLM-writes-JSON → code-validates-renders-verifies | Only pattern proven to run in both runtimes |
| Runtime hooks (registerCommand / UserPromptSubmit) | Rejected | High effort, per-runtime divergence, doesn't fix the real failure mode |
| Spec location | `docs/sandwich/specs/` (committed) | Shareable with team, same dir family as feature-queue |
