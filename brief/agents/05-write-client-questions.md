# Write Client Questions

You receive a JSON context:
```json
{
  "mode": "new" | "refine" | "answer",
  "requirements": { ... },
  "existingClientQuestions": "<string or null>",
  "existingPrd": "<string or null>",
  "existingTechnicalNotes": "<string or null>"
}
```

Write `client-questions.md` — the prioritized list of questions the team needs answered before development starts.

## What makes a good question

- **Blocking:** Cannot design or build something without the answer
- **Clarifying:** The requirement is ambiguous and the answer changes what we build
- **Scoping:** The requirement could mean a small thing or a huge thing; we need to know which
- NOT: "What color should the button be" — that's a designer's job
- NOT: "Can you explain your business in general" — that's what the brief is for

## Structure

```markdown
# [Project Name] — Client Questions

> Generated by sandwich/brief · [mode: new|refine|answer]
> [N] questions · [N answered, if refine/answer mode]

## Priority 1 — Must Answer Before Design

### Q1. [Short question title]
**Context:** [Why we're asking — what we can't design without this]
**Question:** [The exact question to ask the client]
**Impact if unanswered:** [What gets blocked]

...

## Priority 2 — Must Answer Before Development

### Q[N]. [Short question title]
...

## Priority 3 — Nice to Have Before Launch

### Q[N]. [Short question title]
...

---
## Answered *(only present in refine/answer mode)*

### Q[N]. [Short question title] ✓
**Answer:** [Client's answer]
**Resolution:** [What changed in the artifacts as a result]
```

## Rules

- Prioritize questions by how much they block the next phase (design → dev → launch)
- Maximum 5 questions per priority level — force yourself to rank
- If mode is `answer`: move answered questions to the Answered section, mark them with ✓
- If mode is `refine`: keep unanswered questions, add new ones from the new input

Output the complete `client-questions.md` as plain markdown. Nothing else.
