# Write Technical Notes

You receive a JSON context:
```json
{
  "context": { "mode": "...", "hasCodebase": bool },
  "requirements": { ... },
  "existingTechnicalNotes": "<string or null>"
}
```

Write `technical-notes.md` — the tech lead's architecture notes. Be opinionated. Pick a recommendation for every decision — don't just list options.

## Structure

```markdown
# [Project Name] — Technical Notes

> Generated by sandwich/brief · mode: [mode]

## Complexity Assessment
[2-3 sentences: is this standard, or are there genuinely hard parts?]

## Current State Assessment
*(Only for brownfield/refine modes — skip entirely for greenfield)*

| Module | Status | Notes |
|--------|--------|-------|
| [Module] | exists / partial / broken | [what we know] |

**Tech debt signals:** [anything from git history or code that signals debt]

## Architecture Observations
[Freeform: system shape, real-time needs, monolith vs services, etc.]

## Key Technical Decisions

| Decision | Options | Recommendation | Why |
|----------|---------|----------------|-----|
| ... | ... | ... | ... |

## Risky Areas

### [Risk Name]
**Why it's risky:** [explanation]
**Mitigation:** [what to do]

## Suggested Tech Stack *(greenfield/refine only)*

| Layer | Recommendation | Notes |
|-------|----------------|-------|
| Frontend | ... | ... |
| Backend | ... | ... |
| Database | ... | ... |
| Infrastructure | ... | ... |
| Auth | ... | ... |

## Open Technical Questions
- [ ] [Question that must be answered before architecture is final]
```

## Rules

- Brownfield: Current State Assessment section is mandatory
- Greenfield: skip Current State, include full Tech Stack table
- AI/ML modules always get their own Risky Area entry
- In refine/answer mode: mark changed sections with `<!-- updated -->`

Output the complete `technical-notes.md` as plain markdown. Nothing else.
