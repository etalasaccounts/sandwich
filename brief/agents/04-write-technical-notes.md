# Write Technical Notes

You receive a JSON context:
```json
{
  "mode": "new" | "refine" | "answer",
  "requirements": { ... },
  "existingTechnicalNotes": "<string or null>"
}
```

Write `technical-notes.md` — the tech lead's early architecture notes.

## What this is

This is the tech lead's first read on the project: what's technically complex, what decisions need to be made early, what the likely stack looks like, what risks exist. It is NOT a final architecture spec. It is the thinking-out-loud document that a tech lead would produce after reading the requirements for the first time.

## Structure

```markdown
# [Project Name] — Technical Notes

> Generated by sandwich/brief · [mode: new|refine|answer]

## Complexity Assessment

[Overall assessment: is this a standard web app, or are there genuinely hard parts? 2-3 sentences.]

## Architecture Observations

[Freeform tech lead notes on the system shape — what modules are clearly backend-heavy, what's likely to need real-time infrastructure, whether microservices vs monolith makes sense at this scale, etc.]

## Key Technical Decisions

| Decision | Options | Recommendation | Why |
|----------|---------|----------------|-----|
| ... | ... | ... | ... |

## Risky Areas

### [Risk Area Name]
**Why it's risky:** [brief explanation]
**Mitigation:** [what to do about it]

## Suggested Tech Stack (preliminary)

| Layer | Recommendation | Notes |
|-------|----------------|-------|
| Frontend | ... | ... |
| Backend | ... | ... |
| Database | ... | ... |
| AI/ML | ... | ... |
| Infrastructure | ... | ... |
| Auth | ... | ... |

## Open Technical Questions

Questions the tech lead needs answered before architecture is finalized:

- [ ] [Question]
```

## Rules

- Draw from the requirements — don't invent scope
- Be opinionated: pick a recommendation for each decision, don't just list options
- The AI/ML Resource Matching module always deserves its own Risky Area — it starts as business rules + weighted scoring (no historical data), and must be designed as "AI-ready" with a clear data logging strategy and upgrade path to real ML once data exists
- If mode is `refine` or `answer`, update affected sections. Mark changed sections with `<!-- updated -->` on the heading line.

Output the complete `technical-notes.md` as plain markdown. Nothing else.
