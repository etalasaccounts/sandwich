# Write User Flows

You receive a JSON context:
```json
{
  "mode": "new" | "refine" | "answer",
  "requirements": { ... },
  "existingUserFlows": "<string or null>"
}
```

Write `user-flows.md` — narrative user flows, one per major actor journey.

## What this is NOT

This is not a UI wireframe. It is not a clickthrough spec. It is Jeff Patton's "narrative before the map" — the backbone story of how each actor achieves their goal using the system.

## Structure

```markdown
# [Project Name] — User Flows

> Generated by sandwich/brief · [mode: new|refine|answer]

## [Actor Name] Flows

### [Flow Name]: [Goal]

**Trigger:** [What causes this flow to start]

**Steps:**
1. [Actor does something]
2. [System responds]
3. ...

**Outcome:** [What the actor has achieved]

**Exceptions:**
- [Edge case or failure]: [How it's handled]
```

## Rules

- Write one flow per major user goal, not one per page/screen
- Steps alternate between actor action and system response
- Keep flows at the "what" level — not "clicks the blue button" but "requests asset collaboration"
- For a logistics marketplace: cover at minimum — company registration & verification, asset management, marketplace search & collaboration request, dashboard monitoring, API partner integration
- If mode is `refine` or `answer`, update affected flows. Mark changed flows with `<!-- updated -->` on the `###` heading line.

Output the complete `user-flows.md` content as plain markdown. Nothing else.
