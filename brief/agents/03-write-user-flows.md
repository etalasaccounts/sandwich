# Write User Flows

You receive a JSON context:
```json
{
  "context": { "mode": "...", "hasCodebase": bool },
  "requirements": { ... },
  "existingUserFlows": "<string or null>"
}
```

Write `user-flows.md` — narrative user flows. This is Jeff Patton's "narrative before the map" — the backbone story of how each actor achieves their goal.

## Structure

```markdown
# [Project Name] — User Flows

> Generated by sandwich/brief · mode: [mode]

## [Actor Name] Flows

### [Flow Name]: [Goal]
> Status: `planned` | `exists` | `partial`

**Trigger:** [What starts this flow]

**Steps:**
1. [Actor action]
2. [System response]
3. ...

**Outcome:** [What the actor achieved]

**Exceptions:**
- [Edge case]: [How handled]
```

## Rules

- One flow per major user goal — not one per page or screen
- Steps alternate actor action / system response
- Stay at "what" level — "requests collaboration" not "clicks blue button"
- Mark each flow's status based on module status in requirements:
  - `planned` — not built yet
  - `exists` — working in codebase
  - `partial` — started but incomplete
- For brownfield: only write flows for modules that exist or are partial — do not invent flows for things not in the codebase
- In refine/answer mode: mark changed flows with `<!-- updated -->` on the `###` line

Output the complete `user-flows.md` as plain markdown. Nothing else.
