# Write PRD

You receive a JSON context:
```json
{
  "mode": "new" | "refine" | "answer",
  "requirements": { ... },
  "existingPrd": "<string or null>"
}
```

Write a `prd.md` — the canonical product requirements document for the project.

## Structure

```markdown
# [Project Name] — Product Requirements Document

> Generated by sandwich/brief · [mode: new|refine|answer]

## Overview
[2-3 sentences: what is this product, who is it for, what core problem does it solve]

## Actors

| Actor | Role |
|-------|------|
| ... | ... |

## Modules & Features

### [Module Name]
[One sentence description of the module's purpose]

**Features:**
- [Feature name] — [one sentence description]
- ...

### [Next Module]
...

## Integrations
- [Integration name] — [what it connects to and why]

## Constraints
- [Constraint] — [source/reason]

## Stakeholders
- [Name/role] — [their authority or involvement]

## Timeline
[Timeline information if available, else "Not specified"]

## Open Questions
[Count] questions remain — see `client-questions.md`
```

## Rules

- Use the client's own terminology for module and feature names
- Do not invent features not mentioned in the requirements
- Do not recommend a tech stack — that is `technical-notes.md`'s job
- If mode is `refine` or `answer`, update the existing content — don't start blank
- Mark changed sections with `<!-- updated -->` HTML comment on the same line as the heading when in refine/answer mode

Output the complete `prd.md` content as plain markdown. Nothing else.
